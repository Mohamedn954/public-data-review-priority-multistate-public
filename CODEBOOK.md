# Codebook

This repository consolidates de-identified facility-level data already published in the two companion repositories, plus one new file this manuscript itself contributes. Field-level definitions for the underlying facility data are documented in full in each companion repository's own codebook; this file covers what is specific to this repository.

## `anonymized_data/washington/`

Copied unmodified from the Washington companion repository's `anonymized_data/`. See that repository's `README_ANONYMIZED_DATA.md` for full field definitions. Key files used by `scripts/verify_multistate_numbers.py`:

- `WA_AFH_3County_Enriched_ANON.csv` — facility-level roster with `facility_id`, `County`, inspection/investigation/enforcement/stop-placement counts, and `cluster_id` where applicable. 3,457 rows.
- `WA_Operator_Clusters_ANON.csv` — one row per shared-contact operator cluster.
- `WA_RedFlag_Testability_Matrix_v2.csv` — Washington's original five-way testability coding for all 20 RF indicators.

## `anonymized_data/oregon/`

Copied unmodified from the Oregon companion repository's `anonymized_data/`. See that repository's `CODEBOOK.md` for full field definitions. Key files:

- `OR_Providers_Enriched_ANON.csv` — facility-level roster with `facility_id`, `County`, `phone_shared_count`, and enforcement/investigation counts. 792 rows.
- `OR_RF_Evidence_Matrix.csv` — Oregon's eight-tier evidentiary classification for all 20 RF indicators.

## `anonymized_data/combined_evidence_matrix.csv`

**New to this repository.** Generated by `scripts/build_combined_matrix.py` from the Washington and Oregon evidentiary status columns. The Washington and Oregon status labels (`WA_Status`, `OR_Status`) describe whether public data could in principle support an indicator, not whether this study actually computed or assessed it, so `Combined_Status` is not derived from those labels directly. Each row's category was instead audited against the underlying status notes describing what was actually done in each state (`WA_RedFlag_Testability_Matrix_v2.csv`'s `Status_Notes` column and `OR_RF_Evidence_Matrix.csv`'s `Basis` column), assigned to one of seven canonical evidentiary categories per state, then mapped to the most directly evidenced category available across the two states. The audit rationale for every row is recorded as a comment in `scripts/build_combined_matrix.py` itself. This produces the heterogeneous evidentiary accounting reported in this manuscript's Section 3.4 (Indicator classification) and rendered as a table in Supplementary Appendix B. Regenerate with:

```bash
python3 scripts/build_combined_matrix.py
```

| Column | Description |
|---|---|
| `RF_ID` | RF-01 through RF-20, matching both source matrices |
| `Indicator` | Indicator name, taken from Oregon's matrix (wording is materially identical across both source matrices) |
| `WA_Status` | Washington's original five-way testability status for this indicator |
| `OR_Status` | Oregon's eight-tier evidentiary status for this indicator |
| `Combined_Status` | The audited Combined Evidentiary Classification: one of Directly Operationalized, System-Level Evidence, Case-Level Evidence, Partially Operationalized, Contextually Assessed, Publicly Testable Not Completed, or Internal Data Required (RF-15's cell additionally appends "; Not Established" as a display-only refinement; it still counts toward Internal Data Required) |

## Field-status correction: `Has_Public_Reports` (discovered 2026-07-16, during Phase B verification)

The `Has_Public_Reports` column, present in both `WA_AFH_3County_Enriched_ANON.csv` and `OR_Providers_Enriched_ANON.csv`, **is not populated** in the released anonymized files (empty string for every row, both states). This was discovered while building `scripts/build_observability_analysis.py`, which initially used this field to reproduce the manuscript's existing 37.2% Washington "no public record" figure (Supplementary Appendix F) and got 100% for both states as a result — an obviously wrong number that triggered the investigation below, rather than being silently accepted.

**Root cause**: the manuscript's published 37.2% figure was never actually computed from `Has_Public_Reports` — it was computed from `n_docs_total == 0`. `Has_Public_Reports` appears to be a field carried over from an earlier stage of the companion repositories' data pipeline that was superseded by `n_docs_total` as the operational "any record" indicator, without the empty column being removed.

**Resolution**: `scripts/build_observability_analysis.py` uses `n_docs_total == 0` (confirmed exact reproduction of the published 37.2% figure) and documents this in its own source comments. `Has_Public_Reports` is treated as **deprecated** for both states in this repository going forward — it should not be relied on by future scripts or appendices. If a boolean "has any public record" field is needed again, it should be rebuilt deterministically as `n_docs_total > 0` rather than resurrecting the stale column.

This is recorded here as a provenance event, not silently patched: the verification process caught and repaired a real ambiguity in the data before it could propagate into a new appendix.

## `anonymized_data/sensitivity_analysis_detailed.csv` / `_summary.csv`

**New to this repository.** Generated by `scripts/build_sensitivity_analysis.py`, which sweeps the shared-contact clustering rule across thresholds {2, 3, 4, 5}, linkage type (phone/address/combined), county, and facility type, for both states. Fully publicly reproducible from `anonymized_data/washington/WA_AFH_3County_Enriched_ANON.csv` and `anonymized_data/oregon/OR_Providers_Enriched_ANON.csv` alone. Regenerate with:

```bash
python3 scripts/build_sensitivity_analysis.py
```

| Column (summary file) | Description |
|---|---|
| `state` | Washington or Oregon |
| `threshold` | Minimum shared-contact count to qualify a facility as clustered |
| `linkage_type` | `phone`, `address`, or `combined` |
| `facility_type` | Blank for the state-level row; AFH/ALF/RCF/NF for Oregon's per-type stratification (Washington is single-type and has no stratified rows) |
| `n_qualifying_facilities` / `n_total_facilities_in_stratum` / `pct_qualifying` | Counts underlying this manuscript's Section 3.4 threshold-sensitivity findings and Supplementary Appendix A |

## `anonymized_data/corroboration_analysis_detailed.csv` / `_summary.csv`

**New to this repository.** Generated by `scripts/build_corroboration_analysis.py`, which merges Washington's `WA_Operator_Clusters_Corroboration_ANON.csv` (companion repository) with Oregon's new `anonymized_data/oregon/A5_OR_Cluster_Corroboration_ANON.csv` (this repository, Phase A5) into one cross-state cluster-corroboration table: corroboration rate per state, Oregon's evidence-class breakdown, and enforcement/investigation prevalence inside vs. outside corroborated clusters. The script's own docstring documents that Washington's source data cannot make the openly-affiliated-vs-registry-corroborated distinction Oregon's five-class coding can, via an explicit `WA_METHOD_TO_EVIDENCE_CLASS` mapping. Fully publicly reproducible from released anonymized files. Regenerate with:

```bash
python3 scripts/build_corroboration_analysis.py
```

## `anonymized_data/oregon/A5_OR_Cluster_Corroboration_ANON.csv`

**New to this repository (Phase A5).** One row per Oregon shared-contact cluster (49 rows), reporting `cluster_id`, `n_facilities`, `facility_type`, `shared_contact_type`, `evidence_class` (one of: Same openly disclosed brand; Same officers/owners/managers; Same residential or business address; Same legal entity; No public connection found; Indeterminate), `relationship_type`, `corroboration_result` (Corroborated/Not corroborated/Indeterminate), `confidence`, and `uncertainty_note`. Produced by manually checking each cluster against Oregon's public Secretary of State business registry (`data.oregon.gov/resource/tckn-sxa6.json`). The raw registry lookups (business names, addresses, registered-agent identities) underlying this classification are privacy/manual-research-constrained and are **not** part of the public replication package; they are retained as a private research log (`A5_OR_CLUSTER_LOOKUP_LOG_PRIVATE.csv` and related files in the sibling Oregon companion repository's `private/` directory). This file's aggregate classifications are independently verifiable against `anonymized_data/oregon/OR_Operator_PhoneClusters_ANON.csv` (same 49 cluster IDs) but the underlying registry search cannot be independently rerun without re-conducting the manual lookups.

## `anonymized_data/linkage_robustness_detailed.csv` / `_summary.csv`

**New to this repository. Privacy-constrained — not publicly re-runnable at the record level.** Generated by `scripts/build_linkage_robustness.py`, which reads the *raw, non-anonymized* enriched facility files from the sibling private companion repositories (`AFH_RAW_ENRICHED` / `OR_RAW_ENRICHED` environment variables, defaulting to paths inside `PRIVATE-public-data-review-priority-afh/` and `PRIVATE-public-data-review-priority-oregon/`) to test alternative phone/address normalization variants. The primary variant, `phone_key_upstream (PRIMARY)`, reads the raw file's own `phone_key` column, which deliberately excludes one Washington phone number shared by 3 Spokane facilities as a likely generic contact number (documented in `PROVENANCE_MAP.md` and in this script's own source comments — this is the resolution to the WA 53-vs-50 facility-count discrepancy found during Phase B verification). All other variants (`phone_digits_only_no_upstream_exclusion (SENSITIVITY)`, `phone_digits_only_excl_generic (SENSITIVITY)`, `phone_digits_only_cleaned (SENSITIVITY)`, `address_strict`, `address_lenient`) are labeled SENSITIVITY and are not the manuscript's primary construction. The script exits cleanly (code 0) if the raw files are not found, and its outputs contain only aggregate counts, verified via regex pattern search to contain no phone-number-shaped strings. This script cannot be run against the public package alone; its release consists of this processing code, the normalization rules it implements, and its aggregate output files. Regenerate (requires access to the private sibling repositories) with:

```bash
AFH_RAW_ENRICHED=/path/to/WA_AFH_3County_Enriched.csv \
OR_RAW_ENRICHED=/path/to/OR_Providers_Enriched.csv \
python3 scripts/build_linkage_robustness.py
```

## Supplementary Appendix G (RF Specification Registry) and Appendix H (Federal/External Corroboration Crosswalk)

**New to this repository (Phase C).** Both are hand-authored markdown tables in `Hussein_Multistate_Manuscript_Supplementary.md`, not CSV-generated, because they consolidate judgment calls (source tiering, independence classification, disposition) documented in the private `derivation_audit/` research log rather than a mechanical computation. Appendix H's Tables H1-H2 are the public-facing summary of the sources listed in `SOURCE_MANIFEST.md`'s Phase A additions; the full dated search protocol, session log, and source-screening log behind them (`derivation_audit/A2_Corroboration_Search_Protocol.md`, `A3_Search_Session_Log.csv`, `A3_Source_Screening_Log.csv`, `A4_CORPUS_ADJUDICATION.csv`, and related files) are retained as a private research log and are not part of the public replication package, since they were the mechanism for producing Appendix G/H's conclusions rather than facility-level data requiring independent reproduction.

## `anonymized_data/cluster_enforcement_exploratory_detailed.csv` / `_summary.csv`

**New to this repository.** Generated by `scripts/build_cluster_enforcement_exploratory.py`. An exploratory, hypothesis-generating comparison of public enforcement/investigation prevalence between facilities in a shared-contact cluster and facilities in none, computed crude and Mantel-Haenszel-adjusted (stratified by County for Washington, by facility type for Oregon), run both on all facilities and restricted to facilities with `n_docs_total > 0`. This is explicitly not a predictive-validity test (Section 2.6 of the manuscript states predictive validity as untested throughout); it illustrates the same population-validity distinction Section 3.4 documents for the clustering threshold itself, reported in Supplementary Appendix A. `_detailed.csv` has one row per stratum per comparison (exact 2×2 counts); `_summary.csv` has one row per comparison with crude and Mantel-Haenszel-adjusted risk differences, risk ratios, and odds ratios, each with a 95% CI. Regenerate with:

```bash
python3 scripts/build_cluster_enforcement_exploratory.py
```
